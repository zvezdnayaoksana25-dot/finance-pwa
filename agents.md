# AGENTS.md — инструкции проекта Finance PWA

## Назначение проекта

Finance PWA — production-ready PWA-приложение для управления личными финансами. Приложение публикуется на GitHub Pages, работает на iPhone как установленное web-приложение и синхронизирует данные пользователя с его собственной Google Таблицей через Google Drive API и Google Sheets API.

Пользователь должен иметь возможность:

- быстро добавлять доходы, расходы и переводы через центральную кнопку «+»;
- выбирать RUB или USD;
- использовать готовые и собственные категории;
- управлять счетами, картами, наличными, кредитами и накоплениями;
- создавать бюджеты и финансовые цели с конкретными датами и суммами;
- видеть статистику и балансы в рублях и долларах;
- работать без интернета;
- авторизоваться через Google на любом устройстве;
- автоматически синхронизировать данные с Google Sheets;
- редактировать и удалять все созданные сущности.

## Обязательные инструкции для разработки

1. Делать только полноценную production-функциональность.
2. Не создавать заглушки, mock-функции, fake API, demo-mode, prototype-mode или временные кнопки.
3. Не оставлять неработающие кнопки, пустые экраны, `TODO` вместо логики или ненастоящие данные.
4. Каждая видимая функция должна иметь полностью реализованный пользовательский сценарий.
5. Google OAuth, Google Drive, Google Sheets и Currency API должны подключаться к реальным сервисам.
6. Если функция ещё не реализована, её нельзя выдавать пользователю как готовую: нужно либо реализовать её полностью, либо явно остановить этап и сообщить о блокере.
7. Все изменения проверять автоматическими тестами и ручным smoke-тестом основных сценариев.
8. Не изменять существующие пользовательские файлы без необходимости.
9. Не удалять данные, конфигурации или историю без явного запроса пользователя.
10. После каждого этапа обновлять документацию и проверять критерии выхода из этапа.
11. Ошибки сети, OAuth, Google API, Currency API и синхронизации должны обрабатываться понятными сообщениями, retry/backoff и сохранением локальных данных.
12. Не считать локальное сохранение успешным, пока запись не прошла валидацию.
13. Никогда не хранить client secret или refresh token в публичном frontend-коде.
14. Не использовать номер строки Google Sheets как идентификатор записи. Все сущности используют UUID.
15. Удаление данных выполнять через soft delete и синхронизировать удаление между устройствами.

## Принцип работы с данными

Приложение local-first:

- IndexedDB — локальная рабочая база;
- Google Sheets — облачное хранилище и источник восстановления;
- sync queue — очередь исходящих изменений;
- `updatedAt` — версия изменения;
- конфликт разрешается по принципу Last Write Wins;
- удаление хранится через `deletedAt`.

После локального сохранения UI обновляется сразу. При наличии интернета изменение отправляется в Google Sheets. При запуске выполняется двусторонняя синхронизация.

## Доменная модель

Основные сущности:

- `Account` — счёт пользователя;
- `Category` — категория дохода или расхода;
- `Transaction` — доход или расход;
- `Transfer` — перевод между счетами;
- `Budget` — лимит расходов;
- `Goal` — финансовая цель;
- `GoalAllocation` — виртуальное выделение денег на цель;
- `ExchangeRate` — курс валюты с источником и временем получения;
- `SyncQueueItem` — локальное изменение, ожидающее отправки;
- `AppSettings` — настройки пользователя и синхронизации.

Денежные значения хранить в минимальных единицах: копейках и центах. Для каждой операции хранить исходную сумму, валюту, курс на момент операции и рассчитанные значения в RUB/USD. Исторический курс не пересчитывать задним числом.

## Google Sheets

Создавать одну таблицу пользователя со следующими листами:

- `Операции`;
- `Переводы`;
- `Счета`;
- `Категории`;
- `Бюджеты`;
- `Цели`;
- `Распределения целей`;
- `Курсы валют`;
- `Настройки`;
- `Сводка`.

Таблица должна быть читаемой человеком, но приложение использует UUID и заголовки схемы. Любое ручное изменение в таблице должно корректно импортироваться при следующей синхронизации.

## UX-правила

- Главная операция выполняется через центральную кнопку «+».
- Быстрый расход должен добавляться за несколько секунд.
- Сумма — главный элемент формы.
- RUB/USD выбираются без перехода в настройки.
- Готовые категории видны сразу.
- Создание собственной категории доступно из формы операции.
- Все формы имеют состояния loading, success, validation error, network error и retry.
- Все окна должны быть прорисованы и адаптированы под iPhone safe areas.
- Визуальное направление — dark-first Liquid Glass dashboard по референсу: глубокий navy-фон, холодные blue/cyan accents, полупрозрачные карточки, мягкое свечение и чёткая типографическая иерархия.
- Основная сетка desktop: dashboard с двумя колонками; mobile: одна колонка с нижней навигацией и safe-area inset.
- Поверхности используют единые токены радиусов, обводок, blur, теней и focus-ring; не добавлять уникальные ad-hoc стили для отдельных экранов.
- Интерактивные элементы должны иметь touch-target не менее 44px, видимый hover/focus/pressed state и контраст текста WCAG AA.
- Синяя primary-кнопка и центральный FAB — единственные главные акценты; опасные действия используют красный только для ошибки/предупреждения, доходы — зелёный.
- Не менять доменную логику, маршрутизацию, IndexedDB, sync queue, OAuth или API при визуальном рефакторинге. UI-изменения должны быть обратимыми и не менять контракты обработчиков.
- Любой новый экран будущих этапов обязан вписываться в тот же дизайн-контракт: loading, empty, error, success, offline/pending sync и reduced-motion states проектируются одновременно с happy path.
- Liquid Glass использовать с достаточным контрастом и доступностью; декоративное свечение не должно быть единственным способом передать состояние.

## Дизайн-контракт этапов

- Этапы 1–3 считаются функционально готовыми и проходят отдельный UI-polish pass без изменения логики и архитектуры.
- Этап 1: быстрые действия, баланс, операции и счета должны выглядеть как единый dashboard; центральный «+», иерархия суммы и безопасная форма — обязательны.
- Этап 2: бюджеты, цели, фильтры, балансы RUB/USD и empty states используют те же карточки, progress bars, статусы и responsive grid.
- Этап 3: синхронизация, OAuth и ошибки Google показываются теми же notice/card-компонентами с loading, retry и pending states; нельзя скрывать состояние сети декоративным эффектом.
- Этап 4: курсы и аналитика добавляются как визуальные карточки/графики в существующей сетке, сохраняя читаемость реальных данных и валютные подписи.
- Этап 5: PWA, onboarding, настройки и offline UX проходят визуальную проверку iPhone Safari, standalone-режима, safe areas, клавиатуры и reduced motion.
- Этап 6: перед релизом выполняются screenshot/smoke-проверки desktop и iPhone viewport; проверяется отсутствие переполнений, нечитабельного текста и визуально пустых состояний.

## Расширенный референсный scope 2026-07-18

Референс теперь является обязательным продуктовым макетом, а не только визуальным ориентиром. Требуется реализовать 10 связанных экранов: «Обзор», «Быстрый ввод», «Операции», «Счета», «Бюджеты», «Цели», «Статистика», «Категории», «Настройки» и «Синхронизация».

Разрешены ограниченные изменения application/UI-слоя, если без них экран нельзя сделать рабочим:

- введён тип навигационного экрана и единая нижняя навигация;
- экранные представления используют существующий `AppData`, расчёты и репозитории, без дублирования доменной логики;
- создание пользовательской категории оформляется отдельным рабочим сценарием;
- статистика строится на реальных локальных транзакциях, а не на demo-данных;
- кнопки карточек открывают соответствующий рабочий экран или форму; запрещены декоративные кнопки без обработчика;
- существующие формы, UUID, `updatedAt`, `deletedAt`, sync queue, OAuth и Google Sheets contracts нельзя ломать ради пиксельного сходства;
- если референс содержит функцию, которой ещё нет в доменной модели (например, внешний курс валют или архивирование счёта), её нельзя маскировать статическим значением: экран должен показывать честный статус «не подключено/в разработке» либо функция должна быть добавлена полностью.

Для каждого экрана фиксировать responsive states для desktop, iPhone portrait и standalone PWA. Проверять визуальные и функциональные контракты отдельно.

## Тестовая политика

Каждый этап обязан закончиться тестами:

- unit-тесты доменной логики;
- integration-тесты IndexedDB, sync queue и API-адаптеров;
- component-тесты форм и экранов;
- E2E-тесты ключевых пользовательских сценариев;
- regression-проверка уже реализованных функций;
- проверка production build.

Нельзя считать этап завершённым, если тесты отключены, пропущены или заменены ручным предположением.

## Критерий готовности приложения

Приложение считается готовым только когда:

- нет заглушек и demo-mode;
- Google OAuth работает с production-конфигурацией;
- таблица создаётся и синхронизируется;
- операции, переводы, счета, категории, бюджеты и цели полностью редактируются;
- работает офлайн-режим и очередь синхронизации;
- работают RUB/USD и актуальные курсы с кэшированным fallback;
- статистика рассчитывается на реальных данных;
- все кнопки и окна имеют рабочую логику;
- успешны unit, integration, component и E2E тесты;
- production build разворачивается на GitHub Pages;
- проверены iPhone Safari и установленный PWA-режим;
- не осталось `TODO`, `FIXME`, demo-data и временных обработчиков.
